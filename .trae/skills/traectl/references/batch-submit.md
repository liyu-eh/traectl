# 批量提交策略

**不要在同一个任务中混入过多改动。但不是所有任务都需要「单任务单焦点」——简单新增任务可以批量提交。**

---

## 任务分类矩阵

| 任务类型 | 特征 | 提交策略 | 示例 |
|:--------|:-----|:--------|:-----|
| **简单新增** | 独立子命令、纯配置变更、新增文档 | ✅ 批量 2-4 个 | 添加 `version`/`help`/`config` 子命令 |
| **纯删除** | 删文件、删行、无新增逻辑 | ✅ 批量 2-4 个 | 删除未使用的配置项 |
| **纯替换** | str A → str B、不涉及逻辑变化 | ✅ 批量 | 统一改名、改提示文字 |
| **中等修改** | 修改核心逻辑、重构函数、跨文件协调 | 🟡 单任务提交 | 修改 CDP 重连逻辑 |
| **复杂推理** | 架构决策、多文件协调、并发安全 | 🔴 单任务 + 高 timeout | 实现连接池、加锁 |

---

## 批量提交的判断标准

**必须全部满足才能批量：**

| 条件 | 说明 |
|:----|:------|
| 不涉及修改现有核心逻辑 | 只是新增，不是改旧逻辑 |
| 功能间代码隔离 | 不同文件或不同函数，不冲突 |
| 不需要设计权衡 | 实现模式是标准样板 |
| 每项 1-2 句话能说清 | 描述只需要「做什么」不需要「为什么」 |

---

## 批量提交方式

```bash
# 一次性提交多个独立功能
traectl submit "请实现以下 3 个功能：\n1. version 子命令：输出 JSON 版本信息\n2. config 子命令组：get/set/list/export\n3. help 子命令：显示结构化帮助\n\n每个功能独立，不修改现有逻辑，代码隔离在不同文件中。"
```

---

## 并发提交（两个独立任务互不依赖）

当两个任务**完全独立且不修改同一文件**时，可以同时后台提交：

```bash
# 同时发射两个独立任务（Trae CN 会排队处理）
cat task-1.md | traectl submit '任务 1' --no-wait
cat task-2.md | traectl submit '任务 2' --no-wait
```

**适用条件（必须全部满足）：**
1. 两个任务修改不同的文件（无冲突）
2. 两个任务都不依赖对方的结果
3. 都不需要 `--wait` 模式
4. ⚠️ 但串行纪律优先：用户要求一次只能一个任务活动时，禁止并发

---

## 禁忌

| 错误做法 | 原因 |
|:---------|:-----|
| 复杂+简单混打包 | 复杂任务超时后简单任务也丢了 |
| 不检查 Git 状态 | SOLO 可能改乱了工作区 |
| 多个任务在同一会话中不 `new` | 上下文污染 |
| 不等上一个完成就提下一个 | 用户明确禁止过 |
